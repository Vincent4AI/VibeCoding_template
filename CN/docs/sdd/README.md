# 规范驱动开发 (SDD) 流程

## 概述

SDD (Spec-Driven Development) 是一种以规范文档为核心的开发方法论。在编写任何代码之前，先定义模块的接口契约、数据契约和行为规范，然后用规范约束代码实现。

## 生命周期

```
┌─────────┐    ┌─────────┐    ┌──────────┐    ┌─────────┐    ┌──────────┐
│  Draft  │───▶│ Review  │───▶│ Approved │───▶│  Impl   │───▶│ Verified │
└─────────┘    └─────────┘    └──────────┘    └─────────┘    └──────────┘
```

### 状态说明

| 状态 | 含义 |
|------|------|
| **Draft** | 规范初稿，可能不完整，接受大幅修改 |
| **Review** | 规范已基本完成，进入评审阶段，仅接受修正性修改 |
| **Approved** | 规范评审通过，接口已冻结，可以开始实现 |
| **Impl** | 正在实现中，规范不允许再做破坏性修改 |
| **Verified** | 实现完成且通过所有验收标准，模块交付 |

## 开发流程

### Phase 1: 编写规范 (Draft)

1. 从 `templates/module-spec.tmpl.md` 复制模板到 `specs/` 目录
2. 填写所有章节，重点关注：
   - **接口契约**: 所有公开的函数/方法签名、参数类型、返回值
   - **数据契约**: 结构体/类型定义、数据库 schema
   - **行为规范**: 前置条件、后置条件、不变量
   - **错误契约**: 可能返回的所有错误类型及触发条件
3. 标注依赖关系：本模块依赖哪些其他模块的规范

### Phase 2: 评审规范 (Review)

#### 评审 Checklist

- [ ] 接口签名完整且无歧义
- [ ] 所有参数和返回值都有类型标注
- [ ] 错误场景全部列举且有处理方式
- [ ] 前置条件和后置条件可测试
- [ ] 性能约束合理且可度量
- [ ] 与依赖模块的接口一致
- [ ] 验收标准具体、可执行

#### 评审通过标准

- 所有 checklist 项目通过
- 无遗留 TODO 或 TBD
- 依赖模块的规范已处于 Approved 或更高状态

### Phase 3: 实现 (Impl)

#### 实现阶段规则

1. **严格按规范实现** — 公开接口必须与规范完全一致
2. **不允许偏离** — 如果发现规范有问题，先修改规范再改代码
3. **先写测试** — 根据规范的验收标准编写测试用例
4. **内部实现自由** — 私有函数和内部结构不受规范约束
5. **同步更新进度** — 实现完成后更新 STATUS.md 和 TODO.md

### Phase 4: 验证 (Verified)

#### 验证 Checklist

- [ ] 所有验收标准对应的测试用例存在且通过
- [ ] 公开接口与规范完全一致
- [ ] 错误处理覆盖规范中列举的所有场景
- [ ] 性能满足规范中的约束条件
- [ ] 代码通过 lint 和静态分析
- [ ] 无 TODO / FIXME / HACK 注释遗留

## 与 AI 协作

### 使用规范约束代码生成

当让 Claude Code 实现一个模块时：

1. **先指向规范**: "请按照 `docs/sdd/specs/xxx.md` 的规范实现这个模块"
2. **验证一致性**: 实现完成后，对照规范检查所有公开接口
3. **测试覆盖**: 确保验收标准中的每个条目都有对应测试

### 规范作为上下文

规范文档是 Claude Code 的核心上下文。好的规范 = 好的代码生成质量。

关键原则：
- 规范越具体，生成代码质量越高
- 接口签名必须完整（参数类型、返回值、错误）
- 示例代码（用法示例）对 AI 理解意图非常有效
- 边界条件和错误场景要显式列出，否则 AI 会忽略

## 文件组织

```
docs/sdd/
├── README.md                    # 本文件 - SDD 流程说明
├── STATUS.md                    # 模块状态矩阵
├── templates/
│   └── module-spec.tmpl.md      # 模块规范模板
├── specs/                       # 各模块的规范文档
│   ├── module-a.md
│   └── module-b.md
└── adr/                         # 架构决策记录
    ├── README.md
    ├── template.md
    └── 001-xxx.md
```
